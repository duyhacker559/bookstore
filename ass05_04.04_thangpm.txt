Architecture and API Documentation - BookStore Microservices  
1. Introduction  
1.1 Background  
Traditional monolithic systems package all modules into one deployable artifact and often use 
one shared database. While this model can be easier at project start, growth introduces 
technical and organizational friction. In a BookStore context, identity, c atalog, cart, order, 
payment, shipping, review, and recommendation evolve at different speeds and have different 
runtime profiles.  
1.2 Problem Statement  
A monolith for BookStore causes:  
* Hard scaling for specific hot paths (for example catalog browsing).  
* Risky deployment because all modules are released together.  
* Tight coupling across teams and code modules.  
* Technology lock -in and lower parallel development throughput.  
1.3 Objective  
The objective is to implement an academic yet realistic microservice architecture that:  
* Decomposes domain capabilities into independent services.  
* Enforces database -per-service boundaries.  
* Exposes API -first REST contracts.  
* Satisfies assignment functional requirements.  
* Runs with Docker Compose and supports service -level testing.  
2. From Monolith to Microservices  
2.1 Monolithic Baseline  
Monolith characteristics relevant to BookStore:  
* Single codebase, single runtime process.  
* Shared database schema.  
* In-process module calls.  
* One deployment pipeline.  
Advantages at small scale:  
* Simple local setup.  
* Easier transaction handling across modules.  
* One place to debug initially.  
2.2 Monolith Limitations  
As scope increases, practical issues appear:  
* Scaling mismatch: checkout traffic and catalog traffic differ; monolith scales all or nothing.  
* Release risk: a change in review module can impact checkout deployment.  
* Team contention: merge conflicts and long integration cycles.  
* Schema coupling: changing one bounded context may break another query path.  
2.3 Microservice Transition Rationale  
Microservices are selected to isolate business capabilities and operational concerns. The 
BookStore domain naturally fits service decomposition because each context (identity, catalog, 
ordering, payment, shipping, review, recommendation) can be modeled wit h explicit API 
contracts and independent persistence.  
3. Design Principles Applied  
3.1 Single Responsibility at Service Level  
Each service owns one business capability:  
* `customer -service` handles customer records.  
* `cart -service` handles carts and cart items.  
* `pay -service` handles payment records.  
This reduces cognitive load and allows focused testing.  
3.2 Database per Service  
No service directly reads another service database. Data sharing is performed by API calls and 
business identifiers (`customer_id`, `book_id`, `order_id`). This supports autonomy and schema 
evolution.  
3.3 API -First Design  
The system exposes explicit REST endpoints for each service. Current version focuses on `GET` 
and `POST` contracts required by assignment scenarios.  
3.4 Loose Coupling  
Dependencies are explicit and narrow:  
* `customer -service` -> `cart -service` (cart auto creation)  
* `cart -service` -> `book -service` (book validation)  
* `gateway` -> downstream services for UI rendering  
3.5 Stateless Services  
Service endpoints are stateless in request processing (no in -memory session dependency). 
Authentication with JWT is listed as principle; current academic implementation can add JWT in 
next iteration.  
3.6 Resilience  
Current code includes basic request timeout in selected calls. Full resilience stack (retry, circuit 
breaker, fallback policy) is proposed in Section 10 as improvement roadmap.  
4. DDD Decomposition  
4.1 Bounded Context Mapping  
The domain decomposition follows assignment guidance:  
* Identity context: `customer -service`, `staff -service`, `manager -service`  
* Catalog context: `catalog -service`, `book -service`  
* Ordering context: `cart -service`, `order -service`  
* Payment context: `pay -service`  
* Shipping context: `ship -service`  
* Review context: `comment -rate-service`  
* Recommendation context: `recommender -ai-service`  
* Access context: `api -gateway`  
4.2 Ubiquitous Language  
 
Core domain terms used consistently in APIs and models:  
* Customer, Staff, Manager  
* Catalog, Book  
* Cart, CartItem, Order  
* Payment, Shipment  
* CommentRate, Recommendation  
4.3 Context Mapping  
Integration style by context:  
* Upstream identity to ordering: customer creation triggers cart creation.  
* Upstream catalog to ordering: cart validates book before insertion.  
* Ordering to payment/shipping: order references downstream operational records.  
4.4 Aggregate and Data Ownership  
Each service controls its aggregate root and persistence:  
* `Cart` aggregate owns `CartItem` child collection.  
* `Order` aggregate remains isolated from payment/shipment storage.  
* `Recommendation` aggregate stores model outputs independent of transactional tables.  
5. Target Architecture Overview  
5.1 System Topology  
* External clients connect to API Gateway.  
* API Gateway exposes user -facing routes and forwards/aggregates downstream data.  
* 11 domain services handle business APIs.  
* Each service has private persistence.  
5.2 Communication Pattern  
Primary pattern:  
* Synchronous REST over HTTP .  
Logical flows represented in diagrams:  
* `cart -service` validates book existence via `book -service`.  
* `order -service` has business dependency to payment and shipping.  
* recommendation and review services use cross -context identifiers.  
5.3 Deployment Pattern  
* Containerized per service with Dockerfile.  
* Orchestrated via docker -compose.  
* Enabl s independent restart and scaling per service.  
5.4 Architecture Quality Attributes  
* Scalability: selective scale -out by service role.  
* Maintainability: bounded codebases per domain.  
* Reliability: service failure impact localized.  
* Evolvability: API contract boundaries support iterative changes.

10. Reliability and Resilience Assessment  
10.1 Current State  
Observed resilience elements:  
* Timeout in selected HTTP requests (`timeout=5` in gateway and customer/cart service calls).  
10.2 Gaps  
* No retry policy for transient downstream failures.  
* No circuit breaker to prevent cascading failures.  
* No message broker for async decoupling in critical workflows.  
* No idempotency key for repeated client submissions.  
10.3 Improvement Plan  
Priority -ordered plan:  
1. Add retries with exponential backoff for safe idempotent GET requests.  
2. Introduce circuit breaker around inter -service calls.  
3. Add request timeout and fallback responses consistently across all services.  
4. Introduce event bus (RabbitMQ/Kafka) for order -created workflows.  
5. Add distributed tracing (OpenTelemetry) and correlation IDs.

## 15. Detailed Scenario Walkthroughs  
This section describes end -to-end behavior for the most important flows. It can be used as 
speaking script for demo video and as evidence for architecture correctness.  
### 15.1 Scenario A - Customer Registration with Automatic Cart Creation  
Preconditions: `customer -service` and `cart -service` are healthy.  
Steps:  
1. Client submits `POST /customers/` with `name` and `email`.  
2. `customer -service` validates payload and inserts new customer row.  
3. `customer -service` sends `POST /carts/` to `cart -service` with `{ customer_id }`.  
4. `cart -service` creates a cart owned by this customer.  
5. `customer -service` returns `201` to caller.  
Postconditions:  
* New customer exists in `customer -db`. 
* New cart exists in `cart -db` with matching `customer_id`.  
Failure notes:  
* In current implementation, cart call result is not wrapped in compensation logic.  
* Improvement option: transactional outbox + event -driven cart creation.  
### 15.2 Scenario B - Add Book to Cart with Book Validation  
Preconditions: Book exists in `book -service`. Cart exists in `cart -service`.  
Steps:  
1. Client submits `POST /cart -items/` with `cart`, `book_id`, `quantity`.  
2. `cart -service` calls `GET /books/` on `book -service`.  
3. `cart -service` checks whether `book_id` exists in response list.  
4. If valid, `cart -service` saves `CartItem`; otherwise returns `404 Book not found`.  
Postconditions: Cart item is stored only when reference integrity (logical) is satisfied.  
Failure notes:  
* `GET /books/` list scan is simple but not optimal at scale.  
* Improvement option: `GET /books/{id}` endpoint and caching.  
### 15.3 Scenario C - Order, Payment, and Shipment Lifecycle  
Current behavior in code base:  
* `order -service`, `pay -service`, and `ship -service` are implemented as separate APIs.  
* End -to-end orchestration is currently represented at architecture level and can be manually 
exercised through API sequence.  
Suggested orchestrated flow (next iteration):  
1. Create order (`status=pending`).  
2. Trigger payment creation.  
3. If payment succeeds, create shipment.  
4. Update order status to `processing` then `completed`.  
Recommended design option: Use async domain events (`OrderCreated`, `PaymentSucceeded`, 
`ShipmentCreated`) to reduce runtime coupling.  
### 15.4 Scenario D - Rating and Recommendation  
Steps:  
1. Customer posts review to `comment -rate-service`.  
2. Recommendation service periodically reads interaction features and writes scores to 
`recommendation` table.  
3. Gateway exposes recommendation list for UI consumption.  
Discussion:  
* This preserves separation between transactional ordering and analytical recommendation.  
* Recommendation model evolution does not require checkout service deployment.  
16. Architecture Decision Records (ADR Summary)  
**ADR -01: Use API Gateway as Single Public Entry** - Decision: External clients access only 
gateway routes.  
* Reasoning: Simplifies client integration and internal endpoint evolution. Enables future 
centralized concerns: auth, throttling, logging.  
* Consequences: Gateway becomes critical component and must be monitored.  
**ADR -02: Database per Service** - Decision: Every service owns private persistence.  
* Reasoning: Avoids tight schema coupling and cross -team migration conflicts.  
* Consequences: Cross -service joins are impossible at DB level and must be composed via 
APIs/events.  
**ADR -03: REST for Initial Integration** - Decision: Use synchronous REST for first academic 
iteration.  
* Reasoning: Simpler implementation and debugging for coursework timeline.  
* Consequences: Higher runtime coupling than event -driven approach. Need resilience controls 
for production use.  
**ADR -04: Keep Services Stateless** - Decision: No server -side user session required by service 
logic.  
* Reasoning: Better horizontal scaling and simpler failover.  
* Consequences: Authentication context should move to JWT token in future iteration.  
17. Performance and Scalability Considerations  
17.1 Read -heavy vs Write -heavy Paths  
* Read -heavy: catalog, books, recommendation queries.  
* Write -heavy: cart updates during campaigns, order and payment during checkout peaks.  
Scaling guidance: Prefer scaling catalog/book independently from payment/shipping. Add 
caching for hot read endpoints.  
17.2 Latency Hotspots  
Potential hotspots:  
* Gateway fan -out views that query many services.  
* `cart -service` validation call to `book -service`.  
Mitigation:  
* Add timeout, retry for idempotent reads, and bounded concurrency in gateway.  
* Introduce service -level metrics (p95 latency, error rate).  
17.3 Data Growth  
High -growth entities: `CartItem`, `CommentRate`, `Recommendation` records as model refresh 
frequency increases.  
Mitigation: Add retention and archival policy. Partitioning strategy for analytics -heavy tables.
